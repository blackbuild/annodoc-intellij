# Domain documentation

Read `CONTEXT.md` before naming domain concepts in issues, tests, designs, or code. Read relevant ADRs under `docs/adr/` before changing a decision they cover.

If these paths do not exist, proceed silently. The `domain-modeling` skill creates them only when a term or durable decision has actually been resolved.

This repository currently uses one context. `CONTEXT.md` is a glossary, not a specification or implementation guide. Use its canonical terms and call out conflicts with existing definitions or ADRs explicitly.
